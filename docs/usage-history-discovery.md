# Phase 5 に向けた利用履歴の事前調査

作成: 2026-07-12
状態: 調査・提案（実装なし）
前提: [ROADMAP.md](../ROADMAP.md) Phase 5 / [pack-design.md](pack-design.md) / [ai-artifact-return-discovery.md](ai-artifact-return-discovery.md)

> **位置付け**: 本文書は Phase 5 に向けた調査・提案の記録であり、エクスポート履歴の現行仕様の正本ではない。確定仕様は [export-events-design.md](export-events-design.md) を参照。

## 1. 目的

Phase 5（消化マップ・対比資料・共起提示）は「Phase 1〜2 で蓄積される利用履歴が効いてくる段階」と ROADMAP に書かれている。本文書は、**現在の DB に実際に何が蓄積されているか**を確認し、Phase 5 の各機能に必要なデータと現状のギャップ、今から記録を始めるべきものを整理する。

## 2. 現状調査: DB に残る利用履歴

| データ | 現状 | Phase 5 での利用可否 |
|---|---|---|
| 資料項目の現在形（pdf_path・pages・position） | `pack_items` にあり | 「今使っている」は分かる |
| 項目の追加日時 | `pack_items.added_at` | あり（現存項目のみ） |
| 項目の最終更新日時 | `pack_items.updated_at` | あり（現存項目のみ、変更内容は不明） |
| 項目の削除履歴 | **なし**（DELETE で消える） | 復元不能 |
| ページ範囲の変更履歴 | **なし**（UPDATE で上書き） | 復元不能 |
| 資料の削除履歴 | **なし**（CASCADE で項目ごと消える） | 復元不能 |
| PDF/MD/JSON エクスポート履歴 | **なし**（`api_export_pack` は何も記録しない） | 復元不能 |
| 単体切り出し（/export-pdf, /export-md, /export-pdf/save）履歴 | **なし** | 復元不能 |
| エクスポート時点の項目構成 | **なし** | 復元不能 |
| どの項目が一緒に使われたか | 現存する資料の構成からのみ推定可 | 過去分は復元不能 |

結論: **利用「履歴」は実質存在しない**。あるのは現在形スナップショット（pack_items）と、現存項目の added_at/updated_at だけ。

## 3. Phase 5 各機能とのギャップ

### 3.1 消化マップ（本ごとの消化率の可視化）

必要データ: 「この本のどのページを、切り出し・エクスポート・資料追加で使ったか」の累積。

- 現状で賄える範囲: 現存する全資料の pack_items を横断すれば「現在資料に入っているページ」は集計できる。**ただし資料を空にした・削除した時点で消化実績ごと消える**。ROADMAP の「実際の利用実績ベース」という定義を満たすには、資料のライフサイクルと独立した実績記録が要る
- 主要な候補イベント: **エクスポート実行**。これは「書き出した」実績であり、「AI に実際に渡した」ことの証明ではない。Phase 5 で消化実績としてどう評価するかは未確定。単体切り出し（/export-pdf 等）も同種の候補

### 3.2 対比資料（複数書籍の記述を並べる資料構成）

必要データ: 過去にどの本のどの範囲を同じ問いで並べたか。これは資料構成そのもの。

- 現状で賄える範囲: 現存資料はそのまま対比資料の素材になる。履歴が要るのは「過去に作って消した対比構成」の再利用で、優先度は消化マップより低い
- エクスポートイベントに項目構成スナップショットが入っていれば、消した資料の構成も遡れる

### 3.3 共起提示（一緒に使われたページの関連付け）

必要データ: 「同じ資料に同時に存在した」または「同じエクスポートに同梱された」ページ範囲の組。

- 現状で賄える範囲: 現存資料の構成から共起は計算できるが、資料を削除・整理するたびに過去の構成が消える。**エクスポートイベント（項目構成つき）は有力な共起シグナル候補**（「実際に一緒に書き出した」組）。その後 AI へ渡したかどうかは判別できない

### 3.4 過去履歴がない場合に復元不能なもの（まとめ）

1. 消化実績の時系列（いつ・何を渡したか）— 消化マップの「実は8割解体済み」判定の根拠
2. 削除済み資料の構成 — 共起・対比の過去分
3. Phase 3C/3D リリースから記録開始までの期間の実績 — **記録開始が遅れるほど空白期間が伸びる**

## 4. 提案: 最小限のイベント記録

### 4.1 方針

- **v1 はエクスポートイベントのみ記録する**。理由:
  - 消化マップ・共起の主要シグナルであり、Phase 4 の成果物紐づけ（スナップショット）とデータが完全に一致する（1 テーブルで両 Phase に効く）
  - 発生頻度が低く（明示的なボタン操作時のみ）、書き込みコスト・DB 肥大の心配が実質ない
  - 項目の追加/削除/範囲変更まで記録すると、デバウンス保存（400ms）由来の中間状態ノイズが大量に入り、集計前のクレンジングが必要になる。編集履歴は「現在形 + エクスポート時スナップショット」で当面代替できる
- 記録は**成功したエクスポートのみ**（失敗・中断は記録しない）
- 記録失敗はエクスポート自体を失敗させない（ベストエフォート）

### 4.2 テーブル案（Phase 4 の export_events と同一。確定版は [export-events-design.md](export-events-design.md)）

```sql
CREATE TABLE export_events (
    id INTEGER PRIMARY KEY,
    exported_at TEXT NOT NULL,            -- UTC ISO8601
    pack_id INTEGER,                      -- 資料削除後は dangling 許容（JOIN しない）
    pack_name TEXT NOT NULL,              -- スナップショット
    profile TEXT NOT NULL,                -- 'standard' | 'chat' | 'notebooklm'
    format TEXT NOT NULL,                 -- 'pdf' | 'md' | 'json'
    items_json TEXT NOT NULL              -- [{pdf_path, title, pages, position}, ...]
);
```

- スキーマ追加は `ensure_pack_schema` と同様の冪等 CREATE で行い、マイグレーション不要
- 単体切り出し（/export-pdf 等）のイベント化は v2 候補（イベント種別列の追加で拡張できるが、v1 では資料エクスポートに絞る）

### 4.3 イベント種別（将来拡張の枠だけ定義）

| 種別 | v1 | 備考 |
|---|---|---|
| 資料エクスポート（pdf/md/json × profile） | **記録する** | items_json に構成スナップショット |
| 単体ページ切り出し | 見送り | 消化マップ精度を上げたくなったら追加 |
| 項目追加・削除・範囲変更 | 見送り | ノイズ過多。現在形で代替 |
| 成果物取り込み（Phase 4） | Phase 4 で追加 | artifacts テーブル側に created_at があるため専用イベントは不要かもしれない |

### 4.4 保存期間

無期限（削除しない）。根拠: ローカル SQLite・1 イベント数 KB・発生は手動操作時のみで、数年使っても MB オーダー。プライバシーはローカル完結で問題なし。将来必要になったら設定画面に手動削除（全消去）を付ける。

### 4.5 記録開始のタイミング（推奨）

**Phase 3C の export API 配線（C-3）と同時、遅くとも 3D 完了まで**に開始する。

理由:

- 記録箇所は `api_export_pack` の成功パス 1 箇所 + テーブル定義のみ（実装コストが最小の時期）
- Phase 3C/3D で chat/notebooklm エクスポートが使われ始める期間こそ、Phase 4（スナップショット紐づけ）・Phase 5（消化・共起）の最初の実データになる。後付けにするとこの期間が空白になる（§3.4-3）
- 実装ステップとしては独立コミットにできる（例: C-6「エクスポートイベント記録」）。UI なし・既存動作への影響なし・失敗してもエクスポートを妨げない

その後の設計判断により、記録開始は Phase 3C の独立ステップ C-6 として確定した。現行仕様は [export-events-design.md](export-events-design.md) を参照。
