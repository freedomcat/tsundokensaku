# 資料棚 UI 情報設計

作成: 2026-07-19 / 状態: 設計（実装は Phase 2 のUI改善タスクとして実施）
前提: [ROADMAP.md](../ROADMAP.md) Phase 2 / [export-profile-naming-review.md](export-profile-naming-review.md)（profile 名 `chapter` 確定後の用語を使用）

## 背景

資料棚は機能追加（資料の永続化 → ZIP一括エクスポート → AI向けエクスポート最適化）を重ねた結果、ツールバーに8つのコントロールがフラットに並び、「AI向けに書き出す」と「PDF一式を書き出す」の違いが初見で分からない状態になった。

違和感の根源はボタン名ではなく分類にある。「書き出す」という1つの動詞に、性質の異なる3つの概念が同居している。

1. **変換して書き出す** — Markdown分冊・章単位PDF。形が変わる。manifest・概算・警告つき
2. **そのまま書き出す** — PDF一式・MD一式。元データを無加工でZIP梱包
3. **バックアップする** — 資料データJSON。成果物ではなく復元用の構成データ（PDF本体を含まない）

また、バックエンドは既にエクスポートプロファイル機構（`standard` / `chat` / `chapter`）へ統合済みだが、UIは機構統合前の概念（AI向け / PDF一式 / MD一式）を引きずっており、設計がUIに追いついていない。

## 設計原則

この文書の最重要の成果は、UIの分類軸の転換である。

- 旧: **利用先中心** — ChatGPT / Claude / NotebookLM という宛先サービスで選ばせる
- 新: **利用者の目的中心** — 「変換して書き出す / そのまま書き出す / バックアップする」で選ばせる

サービス名・「AI」という語は機能名・グループ名に使わず、説明文の利用例にのみ置く（[export-profile-naming-review.md](export-profile-naming-review.md) の方針を機能名だけでなくUI分類にも拡張したもの）。今後機能が増えたときは、この軸で「どこに属する操作か」を判断する。

## 資料棚の責務

資料棚は「問いに答える資料を、組み立てて、渡せる形にする」場所である。利用者の主活動は3つ、従属的な活動は2つ。

| 活動 | 内容 | 位置づけ |
|---|---|---|
| 集める | ページを追加（検索・PDF・章選択から） | 主活動 |
| 整える | 並び替え・ページ範囲編集・項目削除 | 主活動 |
| 渡す | 完成した資料の書き出し | 主活動・到達点 |
| 管理する | 資料の作成・切替・改名・削除 | 従属（前提操作） |
| 守る | バックアップ・復元（JSON入出力） | 従属（日常フロー外） |

現状の問題は、主活動と従属活動が同じ視覚的重み・同じ列に並んでいること。破壊的操作（空にする）が「ページを追加」の近くにあること。

## 利用者の導線

```
資料を作る → ページを追加する → 資料を整える → 書き出す
（管理する）   （集める）        （整える）      （渡す）
```

画面はこの導線の順に操作が視線移動なく見つかるべきで、「守る」（バックアップ）と破壊的操作は導線から外れた低頻度領域に置く。

## 操作の分類（5グループ）

| グループ | 操作 | 頻度 | 性質 |
|---|---|---|---|
| A. 資料の選択・管理 | 資料の選択・新しい資料・名前を変更 | 中 | 前提操作 |
| B. 組み立て | ページを追加（+カード内の並び替え・範囲編集・項目削除） | 高 | 主活動 |
| C. 書き出し | 変換して書き出す2方式・そのまま書き出す2形式（次章） | 高 | 主活動・到達点 |
| D. バックアップ | 資料データを書き出す・読み込む | 低 | 保全 |
| E. 破壊的操作 | この資料を空にする（確定）・資料を削除（配置はPhase 2C決定待ち。次章参照） | 低 | 要分離 |

分類のポイント:

- E を独立させる。「資料を削除」と「空にする」は破壊的操作として同族だが、対象スコープが異なる。「空にする」は**資料の中身**（項目一覧）が対象で資料棚に属し続ける以外あり得ないため確定。「資料を削除」は**資料そのもの**（リソース単位）が対象で、資料一覧（Phase 2C）が資料の管理を担う設計になった場合、移動どころか資料棚から撤去される可能性がある。日常動線から物理的に離す方針は両者共通
- D を「書き出し」の語族から外す。「資料データを書き出す」がC群と同じ列に並ぶことが、書き出し4種の混同を悪化させている
- primary ボタンは1画面に1つ。ROADMAP Phase 3E「AI向け書き出しを資料完成後の主要操作として明示」に従い、書き出し（C群の入口）に一本化する

## 書き出し方式

UIに出す選択単位は profile そのものではなく **「書き出し方式 = (profile, format) の組」** とする。PDF一式とMD一式は同じ `standard` profile の format 違い（standard だけ `primary_format=None` で実行時に format を選ぶ）であり、profile と選択肢は1:1ではないため。

| 書き出し方式 | (profile, format) | 変換 | 説明（利用例はここに置く） |
|---|---|---|---|
| Markdown分冊 | (chat, md) | あり | 上限に合わせて複数ファイルに分割・結合（ChatGPT・Claudeなどに） |
| 章単位PDF | (chapter, pdf) | あり | 大きな項目を章単位のPDFに分割（NotebookLMなどに） |
| PDF一式 | (standard, pdf) | なし | 1項目=1ファイルでそのまま |
| Markdown一式 | (standard, md) | なし | 1項目=1ファイルでそのまま |

- この対応表をUI側の1箇所（現在の `EXPORT_DESTINATIONS` 相当）に持てば、将来の方式追加は配列1行で済み、UI構造は変わらない。`(chat, html)` や `(chat, epub)` のような拡張も自然に収まる
- 現在の chat 2択（ChatGPT / Claude で同一profile）は「Markdown分冊」1択に畳む。宛先名は説明文の例示へ
- ラベル注意: 「Markdown分冊」と「Markdown一式」はどちらもMarkdownであるため、「分冊・章単位=形が変わる」「一式=そのまま」という語の対比と説明文で軸を明示する

## バックアップ

資料データJSON（書き出し・読み込み）は書き出し方式に含めない。API上は `format=json` として同じエンドポイントに同居しているが、UIの分類とAPIの形式パラメータは独立してよい。

- 実体は資料の構成・ページ範囲・並び順のスナップショット（PDF本体を含まない）
- 用途は復元・移行であり、成果物ではない
- UI上は「バックアップ」の語で括り、低頻度領域（管理メニュー）に置く

## UI案（ワイヤーフレーム）

### Phase 2A: 再グループのみ（機能・モーダル不変）

```
┌─ 資料棚 ──────────────────────────────────────────────┐
│ [資料: ▼ 憲法とAI規制    ] [＋新規] [改名] [削除] [⋯管理] │ ← A + Dは⋯へ。削除は2Cまで現状の位置に残す
│                                                        │
│  3冊・12項目・48ページ・約3.2万トークン                  │ ← 統計は1行に集約
│                                                        │
│ [＋ ページを追加]                  [書き出す ▸(primary)]│ ← B / C を左右分離
│                                                        │
│ ┌─ 書籍カード ────────────────────────────────┐        │
│ │ ...（並び替え・範囲編集・削除は現状どおり）    │        │
│ └────────────────────────────────────────────┘        │
└────────────────────────────────────────────────────────┘

[⋯管理] メニュー（低頻度・破壊的の隔離先）:
   資料データを書き出す / 資料データを読み込む   ← D. バックアップ
   ──────────────
   この資料を空にする                           ← E. 破壊的（資料棚スコープ確定分のみ）
```

「資料を削除」（資料そのものの削除）はPhase 2Aでは1段目ツールバーの現状位置に残す。資料一覧（2C）が資料の管理・削除の主担当になるかどうかが未決のため、今動かすと2C決定後に二重に動く・撤去されるリスクがある。2Cの導線決定後、Phase 2Dで最終配置を確定する。

### Phase 2B: 書き出しモーダル統合 — 完了（2026-07-19）

既存の書き出しモーダル（方式選択+概算+警告）に「そのまま書き出す」2形式を加え、ツールバーの書き出しボタンを1つにした。

```
[書き出す] → モーダル:
   ┌ 書き出し方式 ──────────────────────────────┐
   │ ◉ Markdown分冊                              │
   │    上限に合わせて分割・結合（ChatGPT・Claudeなどに）│
   │ ○ 章単位PDF                                 │
   │    大きな項目を章単位に分割（NotebookLMなどに）│
   │ ──────────                                  │
   │ ○ PDF一式        1項目=1ファイルでそのまま    │
   │ ○ Markdown一式   1項目=1ファイルでそのまま    │
   │                                             │
   │ ── この資料の概算 / 警告 ──                  │
   │                    [キャンセル] [書き出す]    │
   └─────────────────────────────────────────────┘
```

区切り線が「変換される / されない」の境界を示す。概算・警告表示が全方式で共通化される。

## 段階的移行（Phase 2A〜2D）

実装は4段階に分ける。順序は依存関係で決めている（手戻り最小）。

- 統計・用語の整理（2A）は資料一覧（2C）の**前提条件**。用語が未統一のまま一覧を作ると、ぶれた用語が2画面に増殖する
- 書き出しモーダルの統合（2B）は自己完結しており、資料一覧の設計に依存しない。影響を受けるのは**入口ボタンの位置だけ**（暫定配置とし、2Dで確定する）
- ヘッダー行（資料select・新規・改名）の再設計は「一覧→開く→棚」の導線決定（2C）に依存するため最後（2D）

### Phase 2A: UI整理

資料一覧がどうなっても不変な改善のみ。

**完了条件**:

- primary ボタンが1つ（書き出し）
- バックアップ（資料データ書き出し/読み込み）と破壊的操作のうち資料棚スコープが確定しているもの（この資料を空にする）が「管理」メニューへ移動
- 「資料を削除」（資料そのものの削除）は現状の位置に残す。配置は Phase 2C の導線決定後、Phase 2D で確定する
- 統計表示が1行サマリ（用語は本文書の用語表に準拠）
- 空状態に検索画面への導線がある
- 既存テスト全通過（維持）

### Phase 2B: 書き出し整理 — 完了（2026-07-19）

**完了条件**:

- ツールバーの書き出しボタンが1つ
- モーダルで4方式（Markdown分冊 / 章単位PDF / PDF一式 / Markdown一式）を選択できる
- `EXPORT_DESTINATIONS` が (profile, format) の対応表になっている
- 旧書き出しボタン（AI向け/PDF一式/MD一式）が削除されている
- 無効化基準が統一されている（資料があれば書き出しボタン有効、空はモーダル内警告でブロック）
- 既存テスト全通過（モーダル関連の追従は可）

### Phase 2C: 資料一覧 — 完了（2026-07-19）

設計 → 実装の順。「一覧→開く→棚」の導線をここで決定し記録する。

**完了条件**:

- 一覧画面で資料の規模（冊数・項目数・ページ数・推定トークン数）を比較できる
- 不要な資料をまとめて削除できる
- 用語・統計表現が2A のサマリと一致している
- 資料棚との導線（一覧→開く→棚 か否か）が決定され本文書に追記されている

#### 導線の決定

- **URL**: `/packs`（資料一覧画面）
- **ナビゲーション**: 「資料一覧｜資料棚｜設定」の3タブ構成へ拡張する
- **画面の役割**:
  - ホーム（`/`）＝検索の入口（既存の資料棚空状態リンク「検索画面で本を探す」が`href="/"`である前提を踏襲）
  - 資料一覧（`/packs`）＝「管理する」活動の主担当（比較・整理・削除）
  - 資料棚（`/workspace`）＝「集める・整える・渡す」活動の主担当（1資料の編集）
- **一覧→開く→棚は必須の一本道にしない**。資料一覧と資料棚は並列画面とし、ナビゲーションから直接どちらへも到達できる。理由: 資料棚の既存パックセレクタ（`ws-pack-select`、Playwright検証済み）を温存し改修・退行リスクを増やさないため、また既にアクティブな資料へ即座に戻りたい日常操作に不要なワンクッションを課さないため
- **一覧から資料棚を開く操作**: 既存の`TsundokuCart.activatePack(packId)`（`POST /api/packs/{id}/activate`、新規API不要）を呼び、成功後に`/workspace`へ遷移する
- **資料棚から資料一覧へ戻る操作**: 資料棚ヘッダーに軽量な戻りリンクを暫定追加する（本格的な配置確定はPhase 2Dのヘッダー再設計で行う）
- **資料削除の主担当**: 資料一覧側に置く（複数選択してのまとめて削除は一覧でしか成立しない操作のため）
- **`ws-pack-delete`（資料棚の単一削除ボタン）**: Phase 2Cでは移動しない。最終配置（「⋯管理」への統合、または資料一覧側への集約）はPhase 2Dの完了条件で確定する
- **まとめて削除の実装方式**: 新規の一括削除APIは追加しない。既存の単一`DELETE /api/packs/{pack_id}`をクライアント側から選択件数分だけ順次呼び出す（`pack_items`は`ON DELETE CASCADE`のため個々の削除は独立して安全に完結する）
- **統計取得API**: 既存`GET /api/packs`（資料棚の資料切替がデバウンス保存完了のたびに呼ぶホットパス）のレスポンス項目は増やさない。資料一覧専用に`GET /api/packs/stats`を新設し、`item_count`・`total_pages`・`estimated_tokens`を返す
- **ホーム画面の旧エクスポートラベル**（`PDF一式を書き出す`/`MD一式を書き出す`、実体は`/workspace`への単純リンク）はPhase 2Bで実体が変わった後の残骸だが、今回は変更しない。ROADMAP Phase 2の別項目「ホーム画面と資料棚画面の役割を整理」で扱う

### Phase 2D: ヘッダー再設計（完了: 2026-07-19）

2C の導線決定を受けて実施し、完了条件を満たした。以下の設計と実装結果を記録する。

**利用者の到達状態**: 資料棚を開いた瞬間に「今どの資料を編集しているか」が一目で分かり、資料一覧への行き来・低頻度操作・ページ追加の2手段が迷わず見つかる。

#### 実装結果

- 資料棚ヘッダーを4つの情報階層（画面種別と資料一覧導線／現在の資料名と管理メニュー／統計／ページ追加の2起点と書き出し）へ再構成した
- 既存の`#ws-pack-select`を、ID・`change`イベント・選択機能を維持したまま主見出し相当に強調した
- 単一の`#ws-management`に新規作成・改名・バックアップ・空にする操作を集約し、「資料」「バックアップ」「危険な操作」の3グループに整理した
- `#ws-pack-delete`を資料棚から完全に撤去し、資料削除は資料一覧の責務として確定した
- 資料一覧への独立した導線をヘッダーに常時表示し、「資料内からページを追加」と「検索画面で本を探す」の2起点も資料の状態にかかわらず常時表示した
- ナビゲーションの資料棚バッジを、重複する項目ではなくdistinct `pdf_path`による真の冊数表示へ統一した
- 狭幅では資料選択欄の親とselectを縮小可能にし、375px幅での横はみ出しを防止した。Playwrightに寸法ベースの回帰テストを追加し、Python全364件・Playwright全16件の通過を確認した

#### ヘッダー構成の決定

ヘッダーは固定の行数ではなく、次の4つの情報階層として構成する。PC幅では複数の階層が同じ行に収まってよいが、スマートフォン幅では意味のまとまりを保ったまま折り返す（`.ws-toolbar`の既存`flex-wrap`機構をそのまま使い、新しいレスポンシブ機構は導入しない）。

1. 画面種別「資料棚」の表示と、資料一覧への独立した導線
2. 現在選択中の資料名（見出し相当）と、低頻度操作の管理メニュー
3. 現在の資料の統計（既存の`ws-count`/`ws-export-preview`表示をそのまま使う）
4. ページ追加の2起点と、書き出し操作

- **資料名**: 既存`#ws-pack-select`をそのまま使う。要素ID・`change`イベント・選択機能は変更しない。資料名を画面の主見出し相当へ視覚的に昇格させるのはCSSでの見た目の調整のみとし、選択の仕組みは一切変えない
- **管理メニュー**: 既存`#ws-management`を単一の`<details>`のまま維持し、次を統合する。**複数の`<details>`へ分割しない**
  - 資料: 新しい資料・名前を変更
  - バックアップ: 資料データを書き出す・資料データを読み込む
  - 危険な操作: この資料を空にする
- **資料の削除（`#ws-pack-delete`）**: 資料棚から完全に撤去する。管理メニューにも含めない。資料棚に代替の削除導線は追加しない。資料削除は資料一覧の責務として確定する（資料一覧側の削除機能・UIは変更しない）
- **資料一覧への導線**: 資料棚ヘッダーに、資料の中身や状態に関係なく常時表示する独立した導線を置く（本文中の埋め込みリンクではなく、ヘッダー要素として扱う）
- **ページ追加の2起点**: 次の2つを、資料が空かどうかに関係なく常時表示する
  - 現在の資料内の本からページを追加する（既存`#ws-add-open`と`openAddModal()`をそのまま使う。検索範囲は変更しない）
  - 検索画面へ移動して新しい本を探す（既存の`/`をそのまま使う。新しい検索処理・モーダルは作らない）
- **書き出し操作**: 階層4に置く。位置はここで確定とし、以降のPhaseで再検討しない

#### ナビゲーション件数バッジの統一（Phase 2Dの範囲に含める）

`static/pack-store.js`の`updateBadge()`が項目数（`bookCount()`）を表示しており、資料棚ヘッダーの統計（真の冊数、Phase 2Cで`list_packs`/`get_pack`を修正済み）と意味が一致していない。Phase 2Dで`updateBadge()`が真の冊数を表示するよう統一する。バッジの表示位置・外観、`templates/base.html`のナビゲーション構造（資料一覧・資料棚・設定の並列関係）は変更しない。

## 用語

| 用語 | 意味 |
|---|---|
| 資料 | 利用者が組み立てたページ集合（内部名はパック） |
| 項目 | 資料内の1つのまとまり（同一PDFの複数項目も可） |
| 書き出し方式 | `(profile, format)` の組。UIの選択単位 |
| profile | 加工方法（`standard` / `chat` / `chapter`） |
| format | 出力形式（`pdf` / `md` / `json`） |
| 変換して書き出す | 分冊・分割・結合など形を変えて出力すること（chat / chapter） |
| そのまま書き出す | 元データを無加工でZIP梱包して出力すること（standard） |
| バックアップ | JSONによる資料構成の保存・復元。成果物ではない |
| 資料一覧 | 作成済みの資料を比較・整理・削除する画面（`/packs`）。資料棚とは並列画面 |
