### `fitz.open()` の処理時間と性能依存要因について
スキャンPDFのファイルサイズや状態が変化した際の `fitz.open()` 処理時間の実測結果は以下の通りです。

1. **約71MBのスキャンPDF**: `fitz.open()` は平均 **14.88 ms** と非常に高速。
2. **約271MBの巨大スキャンPDF**:
   - **初回実行 (ディスクI/O発生時)**: **1475.25 ms**
   - **2回目以降 (OSキャッシュ有効時)**: 平均 **560.84 ms** (Min: 531.06 ms / Median: 549.91 ms / Max: 1475.25 ms)

> [!IMPORTANT]
> **性能依存に関する注意点**
> 性能（オープン・描画時間）は、単にファイルサイズだけに比例するものではありません。
> - **PDF内部構造**: 埋め込まれた画像の圧縮方式、スキャン時の解像度（DPI）、PDF内のオブジェクト数やタグ情報などに強く依存します。
> - **保存場所とインフラ環境**: ホストOSとコンテナ間のマウント（WSL2のディスクI/O特性など）やネットワーク遅延、SSDかHDDかによって初回I/Oオーバーヘッドは大きく変化します（実測でも初回時に約1.5秒の遅延が発生しています）。
>
> したがって、ページをめくるたび、あるいは拡大確認のたびに何度もPDFを開く設計は、特に初回読み込み時のレスポンス悪化を引き起こします。一度に多くの高解像度画像を要求させない制限付きのAPI設計（案2）は不可欠です。

### 操作仕様
1. **ページ選択 (チェックボックスのON/OFF)**:
   - 各カード左上のチェックボックスをクリックすると、資料に追加するページとして選択されます。このアクションでは**高解像度画像の取得（APIリクエスト）は発生しません**。
2. **高解像度確認 (拡大プレビュー要求)**:
   - カード下部に配置された **「拡大プレビュー」ボタン**（または虫眼鏡アイコン）をクリックすると、右ペインの拡大プレビュー領域に対象ページがロードされます。
   - ロード時にはじめて、対象ページのみを対象とした高解像度確認APIが実行されます。
3. **レスポンシブ対応 (狭い画面・モバイル)**:
   - 画面幅が狭いデバイスでは、右ペインを非表示にし、拡大プレビューボタンをクリックした際に画面中央にモーダル（ライトボックス形式）で拡大画像を表示する挙動に切り替えます。

## 5. API仕様

無制限なパラメータ指定や、複数ページの一括高解像度要求によるサーバー過負荷を防ぐため、パラメータ構造を制限します。

### エンドポイント: `GET /pdf-thumbnails` (既存拡張)
- **後方互換性**: パラメータを省略した場合は従来の一覧用サムネイルを返却します。
- **過負荷防止対策**: `size="detail"` の場合、複数範囲指定を禁止し、**単一のページ番号のみ**の要求を受け付けます。

#### クエリパラメータ
- `pdf_path`: (string, 必須) PDFファイルの相対パス。
- `pages`: (string, 必須) ページ指定。
  - `size="thumbnail"` 時: `1-10` や `3,5,7` などの複数ページ指定可能。
  - `size="detail"` 時: `13` のような **単一の数値のみ** 許容。範囲指定やカンマ区切りが含まれる場合は `400 Bad Request` を返します。
- `size`: (string, 任意) 画像サイズプリセット。
  - `thumbnail` (デフォルト): `zoom=0.3`, `quality=70` で処理。
  - `detail`: `zoom=1.0`, `quality=85` で処理。過大なズーム値の自由指定を排除します。

#### レスポンス (JSON)
既存のレスポンス形式を維持します。
```json
{
  "pages": [
    {
      "page": 13,
      "data": "/9j/4AAQSkZJRgABAQEASABIAAD..." // Base64形式
    }
  ]
}
```

## 6. 状態遷移設計

フロントエンド (`pdf-modal.js`) における拡大プレビューの状態遷移モデルです。

### 状態定義
- `Empty`: プレビュー領域が空の状態（初期表示）。
- `Loading(page)`: ページ `page` の高解像度サムネイルをサーバーから読み込み中の状態。
- `Displayed(page, data)`: ページ `page` の高解像度サムネイルが表示されている状態。
- `Error(page)`: 読み込みエラーが発生した状態（再試行ボタンを表示）。

### 状態遷移図
```
                           [初期化]
                              │
                              ▼
                         ┌─────────┐
                         │  Empty  │
                         └────┬────┘
                              │ 拡大プレビュー(page)クリック
                              │ (インメモリキャッシュ未ヒット)
                              ▼
    ┌────────────────►┌───────────────┐
    │ 拡大プレビュー  │ Loading(page) ├────────┐ APIエラー
    │ (同ページ)      └───────┬───────┘        │
    │ 再試行クリック          │                │
    │                         │ API成功        ▼
    │                         ▼            ┌───────────┐
    │                 ┌───────────────┐    │Error(page)│
    │                 │Displayed(page)│    └─────┬─────┘
    │                 └───────┬───────┘          │
    └─────────────────────────┼──────────────────┘
                              │ 別の拡大プレビュー(page_new)クリック
                              ▼
                    ┌───────────────────┐
                    │Displayed(page_new)│
                    └───────────────────┘
```

### 競合制御 (Concurrency Control)
拡大プレビューのボタンが連続してクリックされた場合、非同期の `fetch` リクエストが並行して走ります。応答速度の遅延によって古い要求が新しい要求の後に到着し、画面表示が逆転するバグを防ぐため、以下の制御を行います。

1. **`AbortController` の利用**:
   - リクエスト開始時に、前回実行中の `AbortController` があれば `abort()` を呼び出し、サーバーへの通信を中断します。
2. **リクエストID / トークンの検証**:
   - `fetch` 完了後、レスポンスのページ番号が「現在アクティブに要求されているページ番号」と一致しているか検証し、古い要求のレスポンスデータは無視します。

## 7. キャッシュ設計 (フロントエンド限定)

サーバー側キャッシュおよび長期間のHTTPキャッシュは初回実装対象外とします。

- **JSインメモリキャッシュ**:
  - `pdf-modal.js` の初期化スコープ内に、 `const highResThumbCache = new Map();` (キー: `pageNumber`, 値: `base64Data`) を定義します。
  - モーダルが開いている間（セッション中）、一度ロードした高解像度画像はメモリ内に保持され、再度同じページのプレビューが押された際はAPIを呼ばず即時描画します。
  - モーダルを閉じた際（`resetThumbPanel` 実行時）にキャッシュはクリアされ、不要なメモリリークを防ぎます。

## 9. 実装時の受け入れ条件 (Acceptance Criteria)

本タスクの実装完了を定義する要件リストです。

* **[AC-1] 操作とリクエストの分離**:
  - サムネイルカードのチェックボックスを操作しただけでは、高解像度画像（`size=detail`）のAPIリクエストが一切送信されないこと。
* **[AC-2] 拡大プレビュー要求**:
  - カード下部の「拡大プレビュー」ボタン（または虫眼鏡アイコン）をクリックした際に対象ページの `size=detail` APIが送信され、拡大プレビュー領域（右ペイン）に表示されること。
* **[AC-3] APIの単一ページ制限**:
  - `/pdf-thumbnails?size=detail&pages=13` は `200 OK` となるが、 `pages=13,14` や `pages=13-15` などの複数ページ指定時には `400 Bad Request` を返すこと。
* **[AC-4] フロントエンド・インメモリキャッシュ**:
  - 同一モーダルセッション内において、一度拡大したページを再度プレビューする際は、APIを叩かずに即座に描画されること。モーダルを閉じるとキャッシュが消去されること。
* **[AC-5] 連続クリックに対する競合制御**:
  - 拡大プレビューを複数ページ連続してクリックした際、`AbortController` またはリクエストIDの検証によって古いリクエストの描画処理が中断され、必ず**最後にクリックしたページのプレビュー画像のみ**が表示されること。
* **[AC-6] エラー状態と再試行表示**:
  - APIリクエストがネットワークエラー等で失敗した場合、プレビュー領域に「読み込み失敗」のメッセージと「再試行」ボタンが表示され、クリック時にロードが再実行されること。
* **[AC-7] 巨大PDFを用いた連続操作検証**:
  - 200MBを超える巨大なPDF（例: 約272MB）において、3〜5ページ連続で拡大ボタンをクリックし、表示順序や競合制御の挙動、操作感に問題がないか確認すること。
  - ※この連続操作時に応答性能（1ページ約800msの遅延）が操作性に与える影響が著しく大きいと判断された場合、ボトルネックの改善（サーバーキャッシュ導入や `fitz.open()` 後のPDFオブジェクト再利用など）は**初回実装範囲から除外し、別タスクとして切り出して検討**すること。
