### `fitz.open()` の処理時間と性能依存要因について
スキャンPDFのファイルサイズや状態が変化した際の `fitz.open()` 処理時間の実測結果は以下の通りです。

1. **約71MBのスキャンPDF**: `fitz.open()` は平均 **14.88 ms** と非常に高速。
2. **約271MBの巨大スキャンPDF**:
   - **初回実行 (ディスクI/O発生時)**: **1475.25 ms**
   - **2回目以降 (OSキャッシュ有効時)**: 平均 **560.84 ms** (Min: 531.06 ms / Median: 549.91 ms / Max: 1475.25 ms)

> [!IMPORTANT]
> **性能依存に関する注意点**
> 性能（オープン・描画時間）は、単にファイルサイズだけに比例するものではありません。
> - **PDF内部構造**: 埋め込まれた画像の圧縮方式、スキャン時の解像度（DPI）、PDF内のオブジェクト数やタグ情報などに強く依存します。
> - **保存場所とインフラ環境**: ホストOSとコンテナ間のマウント（WSL2のディスクI/O特性など）やネットワーク遅延、SSDかHDDかによって初回I/Oオーバーヘッドは大きく変化します（実測でも初回時に約1.5秒の遅延が発生しています）。
>
> したがって、ページをめくるたび、あるいは拡大確認のたびに何度もPDFを開く設計は、特に初回読み込み時のレスポンス悪化を引き起こします。一度に多くの高解像度画像を要求させない制限付きのAPI設計（案2）は不可欠です。

## 4. 推奨する画面構成 (UI/UX設計)

「軽量なサムネイル一覧」と「選択した1ページの拡大確認」の役割と操作を明確に分離します。サムネイル一覧は利用可能な画面幅をできるだけ広く使い、高解像度確認は一覧の上に重ねるオーバーレイモーダルで行います。

| |   軽量画像        |  |   軽量画像        |  |   軽量画像        |  | 軽量画像 | |

### 移行方針
- 既存のサイドペイン型プレビュー（例: `.modal-thumb-detail`）は撤去します。
- 拡大表示は新規のオーバーレイモーダル（例: `.modal-thumb-overlay`）に移します。
- サイドペイン前提のDOM・CSS・状態管理は残さず、サムネイル一覧が横幅を広く使える構成へ戻します。
- 既存の `size=detail` API呼び出し、リクエストトークン、キャッシュ処理など再利用可能なロジックは、オーバーレイ専用状態へ移植します。

### 操作仕様
1. **ページ選択と拡大操作**:
   - 操作の競合を避けるため、役割を明確に分離します。
   - チェックボックスクリック: ページを選択・解除します。
   - サムネイル画像ダブルクリック: 対象ページの拡大モーダルを開きます。
   - 虫眼鏡アイコンクリック: 対象ページの拡大モーダルを開きます。これはPC・タッチ端末共通の主操作です。
   - カード背景クリック: 何もしません。
   - カード全体のシングルクリックによるページ選択は採用しません。
   - これにより、シングルクリックとダブルクリックの競合やクリック待ちタイマーによる遅延を回避します。
   - タッチ端末では虫眼鏡アイコンをタップして拡大します。
   - チェックボックスクリックおよびカード背景クリックでは **`size=detail` のAPIリクエストは送信しません**。
2. **高解像度確認**:
   - サムネイル画像をダブルクリックするか、虫眼鏡アイコンをクリックすると、対象ページの拡大モーダルを開きます。
   - ロード時にはじめて、対象ページのみを対象とした `/pdf-thumbnails?size=detail&pages=X` を実行します。
   - 拡大操作によってページの選択状態が意図せず切り替わらないよう、チェックボックス操作と拡大操作をDOMイベント上も分離します。
   - カード内には虫眼鏡アイコンを表示します。見た目は控えめでも、クリック・タップ可能領域は40px程度を確保します。
   - 必要に応じて短い補助テキストのみ表示します。カード下部の大きなボタンは配置しません。
3. **拡大モーダル**:
   - 画面中央にオーバーレイ表示し、背景を暗くして一覧画面の上に重ねます。
   - ページ画像は画面高の80〜90％程度を上限に、可能な限り大きく表示します。
   - 縦長ページ全体を確認できるようにし、表示領域を超える場合はモーダル内でスクロール可能にします。
   - Escキー、背景クリック、閉じるボタンで閉じられるようにします。
   - 左右キーおよび前ページ・次ページボタンで前後ページへ移動できるようにします。
   - 表示時は150〜200ms程度のフェードインと軽いスケール変更を付けます。派手な演出は不要です。
   - 読み込み中、表示成功、読み込み失敗、再試行の状態を持ちます。
4. **レスポンシブ対応 (狭い画面・モバイル)**:
   - サムネイル一覧は画面幅に応じて列数を減らします。
   - 拡大モーダルは画面端に適切な余白を取り、画像領域は縦スクロール可能にします。

### 画質評価方針
現時点では拡大用の `zoom=1.0`, `quality=85` を維持します。以前のサイドペイン型UIでは表示領域が小さく、高解像度化の効果が分かりにくかった可能性があります。まずオーバーレイモーダルで十分な表示サイズを確保して評価し、それでも文字が読みにくい場合のみ、`zoom=1.25` または `zoom=1.5` を別途ベンチマークします。今回の設計書修正では画質パラメータを変更しません。

## 5. API仕様

拡大モーダルに表示する高解像度画像を、必要になった1ページだけオンデマンドで取得します。無制限なパラメータ指定や、複数ページの一括高解像度要求によるサーバー過負荷を防ぐため、パラメータ構造を制限します。

### エンドポイント: `GET /pdf-thumbnails` (既存拡張)
- **後方互換性**: パラメータを省略した場合は従来の一覧用サムネイルを返却します。
- **過負荷防止対策**: `size="detail"` の場合、複数範囲指定を禁止し、**単一のページ番号のみ**の要求を受け付けます。
- **不正ページの扱い**:
  - `pages=""` や空白のみ: `400 Bad Request`
  - `pages="0"`、負数、小数、全角数字、英字、カンマ区切り、範囲指定: `400 Bad Request`
  - `size="detail"` で数値としては正しいがPDFの範囲外のページを指定した場合: `404 Not Found`
  - `size="thumbnail"` では従来通り、範囲外ページは描画結果から無視します。

#### クエリパラメータ
- `pdf_path`: (string, 必須) PDFファイルの相対パス。
- `pages`: (string, 必須) ページ指定。
  - `size="thumbnail"` 時: `1-10` や `3,5,7` などの複数ページ指定可能。
  - `size="detail"` 時: `13` のような **単一の数値のみ** 許容。範囲指定やカンマ区切りが含まれる場合は `400 Bad Request` を返します。
- `size`: (string, 任意) 画像サイズプリセット。
  - `thumbnail` (デフォルト): `zoom=0.3`, `quality=70` で処理。
  - `detail`: `zoom=1.0`, `quality=85` で処理。過大なズーム値の自由指定を排除します。

#### レスポンス (JSON)
既存のレスポンス形式を維持します。初回実装ではBase64 JSONを継続しますが、将来は `size="detail"` の単一画像に限って `image/jpeg` を直接返す別エンドポイント（例: `/pdf-thumbnail-image`）へ切り替える余地を残します。
```json
{
  "pages": [
    {
      "page": 13,
      "data": "/9j/4AAQSkZJRgABAQEASABIAAD..." // Base64形式
    }
  ]
}
```

## 6. 状態遷移設計

フロントエンド (`pdf-modal.js`) における拡大モーダルの状態遷移モデルです。

### 状態定義
- `Closed`: 拡大モーダルが閉じている状態。
- `Loading(page)`: ページ `page` の高解像度画像をサーバーから読み込み中の状態。
- `Displayed(page, data)`: ページ `page` の高解像度画像が表示されている状態。
- `Error(page)`: 読み込みエラーが発生した状態（再試行ボタンを表示）。

### 状態遷移図
```
┌────────┐
│ Closed │
└───┬────┘
    │ 虫眼鏡クリック / サムネイル画像をダブルクリック(page)
    │ キャッシュ未ヒット
    ▼
┌───────────────┐       API成功        ┌──────────────────────┐
│ Loading(page) ├────────────────────►│ Displayed(page,data) │
└───────┬───────┘                      └──────────┬───────────┘
        │ API失敗                                  │ 前後ページ移動(newPage)
        ▼                                          │ または別ページをダブルクリック
┌─────────────┐                                    │
│ Error(page) │◄──────────── 再試行 ───────────────┘
└──────┬──────┘
       │
       │ Esc / 背景クリック / 閉じるボタン
       ▼
┌────────┐
│ Closed │
└────────┘

※キャッシュヒット時は `Closed` または `Displayed(page)` から直接
  `Displayed(newPage, cachedData)` に遷移し、APIリクエストを送信しません。
※親PDFモーダルを閉じる、ページ選択画面を抜ける、PDFを切り替える場合は
  `ParentModalClosed` / `ThumbPanelReset` / `PdfChanged` 相当のリセット処理として、
  overlay close、`AbortController.abort()`、`requestToken++`、`cache.clear()`、
  body scroll unlock、フォーカス復帰を必ず実行します。
```

### 競合制御 (Concurrency Control)
ダブルクリック、前後ページ移動、再試行が連続した場合、非同期の `fetch` リクエストが並行して走ります。応答速度の遅延によって古い要求が新しい要求の後に到着し、画面表示が逆転するバグを防ぐため、以下の制御を行います。

1. **`AbortController` の利用**:
   - リクエスト開始時に、前回実行中の `AbortController` があれば `abort()` を呼び出します。
   - これは主にクライアント側の表示逆転を防ぐための制御です。FastAPI側で同期的な `fitz.open()` やレンダリングが既に始まっている場合、サーバー側の処理負荷を必ず止められるわけではありません。
2. **リクエストID / トークンの検証**:
   - `fetch` 完了後、レスポンスのページ番号が「現在アクティブに要求されているページ番号」と一致しているか検証し、古い要求のレスポンスデータは無視します。
   - 最後に要求されたページだけを表示対象とします。
3. **detail要求の同時実行抑制**:
   - クライアント側では同一PDFの `size="detail"` 要求を1本に絞り、新しいページ要求が発生したら古い要求をabortし、最新要求だけを有効にします。
   - Loading中は同じページへの再要求を抑止し、再試行は失敗状態または明示的な再試行操作に限定します。
   - 巨大PDFでサーバー負荷が問題になる場合、サーバー側のdetail同時実行数制限は別タスクとして検討します。

### モーダル操作
- Escキー、背景クリック、閉じるボタンで `Closed` に戻ります。
- 左右キーおよび前ページ・次ページボタンは、現在表示中または読み込み中のページを基準に前後ページへ移動します。
- 先頭ページでは前ページ操作、最終ページでは次ページ操作を無効化します。
- 拡大モーダル表示中は背景側のスクロールを抑止し、閉じた後は起点となったサムネイルへフォーカスを戻すことを目標とします。

### アクセシビリティ
- 拡大モーダルには `role="dialog"` と `aria-modal="true"` を付与します。
- モーダルタイトル（例: `p.13 拡大プレビュー`）を `aria-labelledby` で関連付けます。
- モーダルを開いたら閉じるボタン、またはモーダル見出しへ初期フォーカスを移します。
- Tab / Shift+Tab はモーダル内で循環させ、背景側へフォーカスが抜けないようにします。
- モーダルを閉じたら、起点となった虫眼鏡ボタンまたはサムネイルへフォーカスを戻します。

## 7. キャッシュ設計 (フロントエンド限定)

サーバー側キャッシュおよび長期間のHTTPキャッシュは初回実装対象外とします。

- **JSインメモリキャッシュ**:
  - `pdf-modal.js` の初期化スコープ内に、 `const highResThumbCache = new Map();` を定義します。
  - キーは `pdf_path`, `pageNumber`, `size` を含めた文字列（例: `${pdfPath}:${pageNumber}:detail`）とし、PDF切り替え時の誤表示を防ぎます。
  - PDFのページ選択画面が開いている間、一度ロードした高解像度画像はメモリ内に保持され、再度同じページを拡大表示する際はAPIを呼ばず即時描画します。
  - 拡大モーダルを閉じただけではキャッシュを破棄しません。これにより、同じページを再度開いたときに即時表示できます。
  - PDFのページ選択画面全体を閉じた時点（`resetThumbPanel` 実行時）、またはPDFパスが変わった時点でキャッシュをクリアし、不要なメモリリークと誤表示を防ぎます。
  - 初回実装から最大20ページのLRU上限を設け、上限を超えた場合は古い項目から削除します。
  - 将来Blob URLを使う場合は、削除時に `URL.revokeObjectURL()` を必ず実行します。

## 9. 実装時の受け入れ条件 (Acceptance Criteria)

本タスクの実装完了を定義する要件リストです。

* **[AC-1] 操作とリクエストの分離**:
  - チェックボックスクリックではページの選択・選択解除のみが行われ、高解像度画像（`size=detail`）のAPIリクエストが一切送信されないこと。
  - カード背景クリックでは何も起きないこと。
  - カード全体のシングルクリックによるページ選択が採用されていないこと。
* **[AC-2] 拡大プレビュー要求**:
  - サムネイル画像をダブルクリックした際に拡大モーダルが表示され、対象ページの `size=detail` APIが送信されること。
  - 虫眼鏡アイコンをクリックした際に拡大モーダルが表示され、対象ページの `size=detail` APIが送信されること。
  - タッチ端末では虫眼鏡アイコンのタップで拡大できること。
  - 拡大操作によってページの選択状態が意図せず切り替わらないこと。
* **[AC-3] APIの単一ページ制限**:
  - `/pdf-thumbnails?size=detail&pages=13` は `200 OK` となるが、 `pages=13,14` や `pages=13-15` などの複数ページ指定時には `400 Bad Request` を返すこと。
  - `pages=0`、負数、小数、全角数字、英字、空白は `400 Bad Request` を返すこと。
  - `size="detail"` でPDF範囲外のページを指定した場合は `404 Not Found` を返すこと。
* **[AC-4] フロントエンド・インメモリキャッシュ**:
  - PDFのページ選択画面が開いている間、一度拡大したページを再度表示する際は、APIを叩かずに即座に描画されること。
  - 拡大モーダルを閉じてもキャッシュは維持されること。
  - PDFのページ選択画面全体を閉じるとキャッシュが破棄されること。
  - キャッシュキーにPDFパスとページ番号が含まれ、PDFを切り替えても別PDFの画像が誤表示されないこと。
  - detailキャッシュは最大20ページに制限され、上限超過時は古い項目から削除されること。
* **[AC-5] 連続クリックに対する競合制御**:
  - 複数ページを高速にダブルクリックまたは前後移動した際、`AbortController` またはリクエストIDの検証によって古いリクエストの描画処理が中断され、必ず**最後に要求したページのプレビュー画像のみ**が表示されること。
  - `AbortController` をサーバー負荷制御として過信せず、クライアント側で同一PDFのdetail要求が多重に有効化されないこと。
* **[AC-6] エラー状態と再試行表示**:
  - APIリクエストがネットワークエラー等で失敗した場合、拡大モーダルに「読み込み失敗」のメッセージと「再試行」ボタンが表示され、クリック時にロードが再実行されること。
* **[AC-7] 巨大PDFを用いた連続操作検証**:
  - 200MBを超える巨大なPDF（例: 約272MB）において、3〜5ページ連続で拡大表示または前後移動し、表示順序や競合制御の挙動、操作感に問題がないか確認すること。
  - ※この連続操作時に応答性能（1ページ約800msの遅延）が操作性に与える影響が著しく大きいと判断された場合、ボトルネックの改善（サーバーキャッシュ導入や `fitz.open()` 後のPDFオブジェクト再利用など）は**初回実装範囲から除外し、別タスクとして切り出して検討**すること。
* **[AC-8] 拡大モーダル操作**:
  - Escキー、背景クリック、閉じるボタンで拡大モーダルを閉じられること。
  - 左右キーと前ページ・次ページボタンでページ移動できること。
  - 先頭ページ・最終ページでは移動できない方向の操作が無効化されること。
* **[AC-9] レイアウト**:
  - サイドペイン型のプレビュー領域が存在しないこと。
  - サムネイル一覧が利用可能な画面幅を広く使って表示されること。
  - 拡大モーダル表示中は背景スクロールが抑止され、閉じた後は操作元へフォーカスが戻ること。
* **[AC-10] アクセシビリティ**:
  - 拡大モーダルに `role="dialog"`、`aria-modal="true"`、`aria-labelledby` が設定されていること。
  - モーダル表示中はTab移動がモーダル内に閉じ込められること。
  - モーダルを閉じると、起点となった虫眼鏡ボタンまたはサムネイルへフォーカスが戻ること。
  - 虫眼鏡アイコンは見た目が小さくてもクリック・タップ可能領域が40px程度確保されていること。
