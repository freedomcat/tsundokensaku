# つんどけんさく ロードマップ

最終更新: 2026-07-20

## コンセプト

- 積読を知識に変える
- AI（NotebookLM や ChatGPT など）へ渡す資料を組み立てる、ローカル完結型ワークスペース

つんどけんさくは AI 要約・ベクトル検索・RAG を内蔵しません（[README「あえて入れていないもの」](README.md#あえて入れていないもの) 参照）。AI はあくまで外部のサービスとして使い、このアプリは **AI に渡す前（資料の組み立て）と、AI から戻った後（知識の定着）** を支えることに徹します。

利用者の基本の流れは「**検索 → 資料に追加 → 資料机で整理 → エクスポート**」です。現行UIの「資料棚」は、今後「資料机」へ名称変更する予定です。ロードマップはこの体験を先に完成させ、そのうえで AI 向けの最適化・帰還へ進みます。

## 情報アーキテクチャ

つんどけんさくの情報アーキテクチャおよび各UI・操作用語は、今後以下の設計方針と名称へ整理・変更される予定です。なお、過去の完了済みフェーズ（Phase 1など）の記述については、実装当時の名称（「資料棚」など）を歴史的記録としてそのまま維持しています。

- **資料一覧**: 作成済みの資料を一覧表示し、選択する場所。
- **資料机**: 選択した資料に本・ページ・検索結果などを集め、並び替え・削除・構成を行う作業空間（現行UIの「資料棚」）。
- **本の登録**: 検索対象となる本やPDFを登録し、インデックス作成・OCR・検索対象の管理などを行う場所（現行UIの「設定」）。
- **資料管理**: 現在開いている資料に対する、新しい資料・名前変更・書き出し・読み込み・この資料を空にするなどの操作をまとめたメニュー（現行UIの「管理」）。
- **資料に追加**: 検索結果から現在選択中の資料へ本・ページを追加する操作。
- **資料机で整理**: 資料へ追加したあと、資料机へ移動し、資料の構成や並び順を整理する導線（現行UIの「資料棚で編集」）。

### Phase 0: 検索基盤 — 完了

「蔵書から探す」は一段落しました。

- 蔵書横断検索: PDF 本文・タイトル・Kindle 本・Scrapbox/Cosense メモ・書籍ノート（FTS5 + Sudachi 分かち書き + trigram 部分一致）

- 出典情報の基礎（追加時点のタイトル・ページ範囲・追加日時）を保存 — 後続 Phase のデータ基盤

## 横断的な品質・運用基盤

機能 Phase とは別に、リポジトリ全体を継続的に保守・公開していくために整備する品質・運用上の観点。個人のローカル利用を中心としつつ、デモ公開や将来の利用者拡大に応じて段階的に対応する。実装順や Phase 番号はここでは確定せず、着手時に個別に計画する。具体的な数値目標・採用ツールは、確定済みのものを除き測定・調査してから定める。

### Must（機能開発と並行して近いうちに整える）

- [ ] **CI での Python テスト常時実行** — 目的: ローカルの手動実行だけに依存しない回帰検知。現状: GitHub Actions（`ci.yml`、pull_request / push 発火の Python テストジョブ）を Phase 4A-1（PR #1）で追加し、develop へ導入済み。PR #1・#4 のマージ時に Actions の成功を確認した。継続運用が定着したと言える段階ではまだない。次: 今後の複数 PR でも安定して緑になることを確認し、必須チェック化を判断する。完了目安: 今後の複数 PR でも安定して必須チェックとして機能する状態
- [ ] **テスト戦略の明文化** — 目的: テスト件数（現在 Python 383 件・Playwright 19 件）ではなく「何をどの層で保証し、何を保証しないか」を記録する。現状: 単体（DB・エクスポートロジック）・HTTP API（TestClient）・E2E（Playwright、`--workers=1` 制約は tests/playwright/README.md に記録済み）の3層が実態としてあるが、保証範囲の文書はない。破損・暗号化・本文抽出不能・巨大・空の PDF、不正メタデータ、ファイルの削除・移動・改名、操作の競合などの異常系は現状ほぼ未検証で、将来の検証候補として扱う。次: docs/ にテスト戦略メモを作り、保証対象と非保証範囲・異常系候補の一覧を記録する。完了目安: 新機能のテストをどの層に書くべきか迷わない状態
- [ ] **データ保全とスキーマ管理の方針決定** — 目的: 資料・エクスポート履歴・AIノート（Phase 4）と、失いたくないデータが増えるため、更新・障害時にデータを失わない手順を持つ。現状: スキーマは冪等な `CREATE TABLE IF NOT EXISTS` 方式でバージョン番号管理はない。資料データの JSON 書き出し・読み込みは実装済み。DB ファイル全体のバックアップは手動コピー頼み。次: スキーマバージョンの記録方法、アプリ更新前のバックアップ手順（自動か手動か）、破損検出時の対応、バックアップ形式の互換性方針を調査・設計する。完了目安: 「更新に失敗しても直前の状態へ戻せる」手順が文書化されている状態

### Should（デモ公開や共同開発を意識した段階で対応）

- [ ] **Playwright テストの CI 実行** — 現状: ローカル実行のみ。CI 化にはクリーン DB（同梱サンプル PDF 3冊のみのインデックス）で全件通ることの検証と、サーバー起動・インデックス構築のジョブ整備が必要。まずローカルでクリーン DB 検証を行ってから CI へ載せる
- [ ] **lint・format チェック** — 現状: 未導入（Python・JS とも）。ツール選定は導入時に判断。目的は差分レビューをスタイル指摘から解放すること
- [ ] **Python 依存関係の固定** — 現状: pyproject.toml は下限指定（`>=`）のみでロックファイルなし（npm 側は package-lock.json あり）。Docker イメージ再ビルド時に依存が変わり得る。ロック方式・開発依存と本番依存の分離・更新方針を調査して決める
- [ ] **セキュリティ検証の体系化** — 現状: パストラバーサル対策（books_dir 境界の解決・保存先の `relative_to` 検証）とデモモード（`DEMO_MODE` での書き込み系無効化）は実装済みだが、これらを狙い撃ちする回帰テストは体系化されていない。役割の異なる3つの手段を組み合わせる。(1) **手動の脅威分析**: 境界突破（シンボリックリンク含む）・ZIP Slip・悪意ある/異常な PDF・巨大ファイルによる資源枯渇・PDF タイトルや資料名経由の XSS・アップロード中断時の一時ファイル・エラーメッセージでのローカルパス露出・SQLite 同時書き込みを検証候補として整理する。(2) **Python でのセキュリティ回帰テスト**: 分析で確認した防御を狙い撃ちするテストを少しずつ足し、退行を防ぐ。(3) **OWASP ZAP による Web UI・HTTP API の Passive/Active Scan**: HTTP 層の継続的な検証手段として、デモ公開前に限らず定期的に流す。ZAP で確認できるのは HTTP 層の一部であり、ファイルシステム境界や PDF 処理はコードレビューと (1)(2) で補う。デモモードで無効化すべき機能の一覧は、新機能追加のたびに監査する（Phase 4A のデモモード制約はその第1号）
- [ ] **性能測定と基準策定** — 現状: 性能目標値はない。100冊・1,000冊規模でのインデックス時間・差分インデックス・検索応答・DB サイズ・メモリ・サムネイル生成・エクスポート時間・大量データ時の UI 応答性を、まず実蔵書で測定してボトルネックを把握し、その結果から基準を定める。数値目標を先に決めない。改善の実施は別項（将来検討）とする
- [ ] **実利用者による検証** — 自動テストでは確認できない UX の検証。開発者本人以外による初回セットアップ、説明なしで「検索 → 資料へ追加 → ページ追加 → エクスポート」を完了できるか、初回インデックスの理解しやすさ、エラーメッセージの分かりやすさ、スマートフォン・小画面での操作性、実蔵書規模での利用観察。結果は README・初回導線・エラーメッセージの改善へ直接反映でき、性能改善より早い段階で価値がある。フィードバックを本ロードマップへ反映する経路もここで決める
- [ ] **ブランチ運用と PR の定着** — 現状: PR #1（Phase 4A-1）・PR #4（Phase 4A-2）を Merge commit 方式でマージ済み。積み上げ PR（PR #4 は base を feature ブランチから develop へ変更後にマージ）、CI 確認後のマージ、feature ブランチの削除まで一連を実施した。PR #2 は Open / Draft のまま。開発正本は develop だが GitHub のデフォルトブランチは master のままで、運用が定着したと言える段階ではまだない。次: デフォルトブランチの整理、必須チェック（CI 緑）とブランチ保護の要否判断（private リポジトリのプラン制約も含めて）、PR 運用ルールの文書化、AI レビューをどの段階で必須とするかの明文化。変更理由とレビュー履歴が GitHub 上で追える状態を目指す

### 将来検討（利用者拡大・公開運用が視野に入った段階で）

- [ ] **型チェック導入** — 現状: 型ヒントは書かれているが検証は未導入。導入コストと効果を見て判断
- [ ] **性能改善の実施** — 測定と基準策定（Should）でボトルネックが判明した箇所への対応。長時間処理の進捗表示・中断・再実行もこの段階で扱う
